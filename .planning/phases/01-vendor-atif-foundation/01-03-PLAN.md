    def test_models_import_cleanly() -> None:

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Negative fixture accidentally validates (validator bug or fixture too lenient) | mitigate | `test_invalid_fixture_rejected` asserts BOTH `validator.validate(...) is False` AND a step_id-related error in `validator.errors`. | T-03-02 | Information Disclosure | `validate()` exposes too much error detail in passthrough mode | accept | LOW — `validate()` returns only bool per D-08. Detailed errors require explicit `TrajectoryValidator()` instantiation, which is a deliberate caller choice. |
| T-03-03 | Denial of Service | Pydantic forward-reference resolution failure at import time (RESEARCH.md Pitfall 4) | mitigate | `test_models_import_cleanly` exercises the full models import chain. Failure surfaces immediately on `pytest -v`, blocking the plan from completing. The acceptance criterion "uv run pytest -v ... reports `5 passed`" gates this. |
| T-03-04 | Tampering | Smoke test passes vacuously if fixture files are missing | mitigate | `_golden_paths()` uses `rglob("*.json")` and excludes `_invalid/`; if Plan 01 didn't deposit the goldens, parametrize would have zero items and pytest would surface `collected 0 items`. The acceptance criterion checks for the literal `5 passed` count. |
| T-03-05 | Repudiation | `validate()` lacks audit logging | accept | LOW — vendored Harbor validator does not log; daydream's `validate()` is a pure passthrough. Audit trail belongs at the recorder layer (Phase 2), not the validator wrapper. |
